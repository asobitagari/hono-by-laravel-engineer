# services

ドメインサービスを配置するディレクトリ。ドメインごとにサブディレクトリを切る。

## useCases との使い分け

| | useCases | services |
|---|---|---|
| 役割 | アプリケーション操作のフロー制御（複数の操作の組み合わせ） | 単一エンティティに収まらないビジネスルールの表現 |
| 例 | ログイン（認証→トークン発行→DB保存） | プロフィール名の変更（名前のみ変更可、メールは変更不可） |

## ここに書くもの・書かないもの

```
# 書く → ビジネスルールが存在する操作
updateProfileNameService   # メールアドレスは変更不可、というルールがある
updatePasswordService      # 現在パスワードの確認が必要、というルールがある

# 書かない → リポジトリを呼ぶだけの単純な操作
getProfileService          # repo.findById() を呼ぶだけ → ルートから直接リポジトリを呼ぶ
```

単純なCRUDにサービスを挟むと、層が増えるだけで意味がない。

## 命名規則

```
{動詞}{対象}Service.ts
```

**対象を具体的に書く。** `updateProfile` ではなく `updateProfileName` のように、何を変更するかを明示する。

## 関数シグネチャの規約

- 第1引数はリポジトリインターフェース（`domain/` で定義）
- HTTPの知識を持たない（`Context` や `Request` に依存しない）
- エラーは例外をスローする

## 新規ファイル追加時のチェックリスト

- [ ] 単純なリポジトリ呼び出しではなく、ビジネスルールが存在する
- [ ] ファイル名が `{動詞}{対象}Service.ts` の形式で、対象が具体的になっている
- [ ] 引数に Hono の型（`Context` など）が含まれていない
